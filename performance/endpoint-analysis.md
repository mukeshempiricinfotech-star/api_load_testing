# API endpoint and supporting-component analysis

Review date: **02 Jul 2026, Asia/Kolkata (UTC+05:30)**

Scope: 15 routable HTTP endpoints in the checked-out `api_load_testing` repository at commit `b8ef7b8` before this run's changes (14 `/api` business endpoints plus `/health`). Owners come from `CODEOWNERS`. Jira Component Mapping uses only mappings evidenced by repository ticket metadata; missing mappings are explicitly recorded instead of invented.

All endpoints pass through the 1 MiB `express.json` parser and the shared JSON 404/error middleware in `src/app.js:10-21`. All route handlers use the `asyncHandler` rejection wrapper. Protected endpoints additionally use the HS256 JWT Bearer-token middleware in `src/middleware/authenticate.js:5-9`. Route-specific Zod validation is noted below.

## Production traffic evidence

`api-inventory.json` records requests/day, average RPS, worst observed peak RPS, worst observed daily P95, and worst observed daily 5xx error percentage for all 14 business endpoints from `performance/traffic-seed.json` (seven complete days ending 01 Jul 2026). That source is labelled `datadog_seed` by the repository. Live Datadog and GA4 connectors and the required signed-in browser runtime were unavailable, so live access could not be independently verified. P99 is recorded as `null` because neither the seed nor an accessible source exposes P99. `DELETE /api/cart/items/:id` has no source measurements and uses the repository's explicit conservative fallback (25,000 requests/day, 10 peak RPS, 250 ms P95, 0.5% errors), marked **Estimated Traffic**. `/health` is also marked **Estimated Traffic** because no telemetry row exists.

## User-journey basis

No request-sequence telemetry, GA4 geography/device split, or peak-hour detail was available. The four journeys in `config/load-profiles.json` are therefore marked **Estimated Journey** and inferred only from the controller/service flow shown above:

1. Authentication lifecycle: register → login → refresh → logout.
2. Browse/Discovery: health → catalogue → search → product detail → reviews.
3. Cart Management: login → browse/detail → get cart → add item → remove item → logout.
4. Purchase/Checkout: login → search/browse → detail → add/get cart → create order → capture payment → list orders → logout.

The matching k6 and Locust files execute the same steps and parameters for each journey. All six load profiles (Smoke, Load, Stress, Spike, Soak, Breakpoint), arrival-rate targets, reserved VUs, ramp/stage timings, and think times are recorded in `config/load-profiles.json`.
